Step 1 Guide

Import: get clean source data into MNO

This step decides what you are feeding into MNO. The goal is simple: start from a valid source and end with a valid MNO memory store.

Use Raw Source Files most common

Pick raw files or a folder when you are starting from conversation exports, logs, or transcripts.

  • Good for reopening an existing memory system.
  • Lets you skip archive import entirely.
  • Still validate before moving on.

Build Style Guide

How strict should episode building be?

This setting only affects how draft episode cards are created. It does not change live chat behavior later.

Strict cleanest draft

Use this when you want fewer cards and less junk. MNO only keeps the strongest, clearest episode candidates.

  • Best for noisy exports or when review time matters.
  • More likely to miss weak or fragmented memories.
  • Good if you want the review queue as small and clean as possible.

Example: “This export is messy. Only show me solid episodes worth keeping.”

Balanced normal default

Use this for most real imports. It keeps a healthy middle ground between recall and cleanup work.

  • Good default for a normal personal archive.
  • Keeps more valid memories than strict without opening the floodgates.
  • Usually the best first pass if you are unsure.

Example: “Give me a solid review draft without drowning me in weak cards.”

Memory Assist highest recall

Use this when you care more about not missing memories than keeping the review queue small.

  • Best for sparse, fragmented, or lightly detailed exports.
  • Creates more draft cards, including weaker candidates.
  • Expect more review cleanup afterward.

Example: “Pull in more maybe-useful memories. I’d rather review extra cards than miss something.”

Simple rule: start with balanced. Move to strict if the draft is too noisy. Move to memory assist if the draft is too thin.

Step 3 Guide

Review: decide what survives into the published memory set

Review is where you clean up the draft. Every card must end in one of three states: approve, edit, or reject.

Made-up example card

Weekend garage cleanup
The draft card says the cleanup is scheduled, but it forgot that the paint cans still need a safe drop-off plan.
actors: you, neighbor | topics: home, errands
How to decide review flow
  • Approve As-Is when the title, summary, and tags are already right.
  • Quick Edit when the card is basically good but needs a title cleanup, better summary, or cleaner tags.
  • Reject when the card is junk, misleading, duplicate, or not worth keeping.

Example edit: keep the card, but add the missing drop-off detail because it changes what you need to remember later.

What moves you forward: once every draft card has an explicit decision, Review is done, the rail turns green, and Publish becomes the next active step.

Every draft card must be approved, edited, or rejected before publish.

Use the card grid for fast decisions. Quick Edit is for cleanup, not for hunting through giant forms.

Step 4

Step 4 Guide

Publish: freeze the reviewed set you actually trust

Publishing takes the reviewed cards and creates the frozen set the runtime should actually use. This is the “real” reviewed memory artifact.

What publish does freeze reviewed set
  • Compiles approved and edited cards into the reviewed set.
  • Ties that set to the current store fingerprint and build.
  • Keeps a restorable history so you can recover a last good copy.
When to republish only after changes
  • If you change review decisions, publish again.
  • If the store changes enough to invalidate the reviewed set, rebuild and review again.
  • Publish is the handoff into Verify.
Publish shows what will be frozen into the reviewed memory set.

Local settings

Chat Memory Guide

How eager should live chat be to use memory?

This setting affects live chat routing. It does not change how episode cards are built.

Balanced recommended

Use the normal routing rules. MNO pulls memory when the message looks memory-relevant and stays quiet when it does not.

  • Best default for most day-to-day chat.
  • Keeps a practical middle ground between recall and over-searching.
  • Leaves explicit memory requests and high-risk cases alone.

Example: “Act normal. Use memory when it helps, but don’t force it into every reply.”

Chat First lowest recall

Prefer normal chat and suppress some memory lookups unless the user is clearly asking for recall.

  • Best when you want fewer memory probes.
  • Useful for casual conversation where memory would feel noisy.
  • Explicit memory requests still go through.

Example: “We’re mostly just chatting. Don’t keep digging through memory unless I clearly ask.”

Memory Assist highest recall

Push the router to use memory more aggressively, especially on weaker or borderline recall signals.

  • Best when you want stronger memory help during live chat.
  • Good for identity, relationship, planning, or continuity-heavy use.
  • Can feel heavier because it searches more often.

Example: “Lean into memory. I’d rather get more recall help than risk missing useful context.”

Important: explicit memory requests and high-risk cases still escalate memory use even if you choose chat first.
